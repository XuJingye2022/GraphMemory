## AI记忆系统方案构想 (v2.0)

> 这是一个关于如何让AI像人一样记住、联想和遗忘的构想方案。不涉及具体技术实现，只描述核心机制。

---

### 一、核心理念

记忆不是简单的文件归档，而是一张**有方向的联想网**。每个知识点都是一个节点，知识点之间的联想就是带方向的边。

**关键洞察**：联想往往是不对称的。
- 想到“下雨”，很容易联想到“地湿”；
- 但想到“地湿”，不一定能立刻想到“下雨”（可能是洒水车）。

这种不对称性正是人类思维的常态。因此，记忆网络中的边**有方向**，但方向不是固定的——用得多了，弱方向也可以变强。

---

### 二、记忆的基本单元

**节点**：一个独立的知识点、事实、事件或技能。例如：“咖啡含有咖啡因”、“登录Gmail的步骤”。

**有向边**：从一个节点指向另一个节点的联想。每条边有两个基本属性：
- **正向强度**：从这个节点想到那个节点的容易程度。
- **反向强度**：从那个节点回想到这个节点的容易程度。

> 初始时，因果关系等天然联想往往正向很强、反向很弱。但如果反向被反复使用，它的强度也会逐渐增加。

---

### 三、记忆的轻重缓急：核心度与注意力的配合

**核心度**是节点自身的属性，独立于任何边。核心度高的节点，就像你脑中常挂在嘴边的知识，更容易被优先加载到当前思维中。

当需要思考或回答问题时，系统会：

1. 根据当前问题和核心度高低，挑选一批最相关的节点。
2. 再沿着它们之间的有向边，找到强联想的邻居节点（正向和反向都可能，取决于你想探索的方向）。
3. 将这些节点加载到**工作记忆**（即当前的思考小剧场）中。

这个过程是有选择性的：**核心度高、关联强度大的节点，更容易被拉进工作记忆**。这就实现了“分清轻重缓急”。

---

### 四、大模型上下文与记忆系统的协作

工作时，通常会把各种记忆提取出来，形成工作记忆。

**工作记忆**：容量极小（就像人一次只能想几件事）。在AI中对应当前正在处理的上下文窗口。  

当需要思考或回答问题时：
1. 先从记忆系统中，根据当前问题和核心度高低，挑选一批最相关的节点。
2. 再沿着它们之间的有向边，找到强联想的邻居节点。
3. 将这些节点加载到工作记忆中，供推理使用。

当完成工作时：
1. 如果是已有的知识点和流程，会加强成功实施的连接和知识点核心度。
2. 如果是没有到知识点和流程，会存入记忆系统。其连结和核心度具备遗忘系统。

---

### 五、遗忘：像人一样渐渐淡忘

每条联想边的强度（包括正向和反向）都会随时间衰减——如果不被使用。

- **早期**：衰减很快。一个新建立的联想，几天不用就几乎消失。
- **反复使用后**：衰减越来越慢。一条被你反复走过的联想路径，会变得“根深蒂固”，甚至可以保持数月或数年。

这就是所谓的“记忆曲线”：刚学的东西忘得快，复习多次后忘得慢。

**核心度的衰减**同样适用：重要的知识点如果不被使用，也会慢慢变得不再重要。

---

### 六、记忆重放：有目的地巩固

除了合并相似节点和剪枝无用节点，系统还应该定期进行**记忆重放**——但不是无差别地回放所有记忆，而是**结合AI自身的身份和职责**，有针对性地挑选那些**与它的核心任务、常用工作流、典型思维链高度相关**的记忆进行加强。

**具体做法**：
- 每个AI或Agent都有明确的**身份定义**（例如“编程助手”、“客服”、“个人秘书”）和**职责范围**（例如“帮助调试代码”、“回答产品问题”、“管理日程”）。
- 在重放时，优先挑选：
    - 与该身份职责**直接相关**的高核心度节点。
    - 那些**被反复使用且成功的工作流**（即一连串有序的节点和边）。
    - 典型的**思维链模板**（例如“遇到X问题时，通常先检查A，再考虑B，最后执行C”）。
- 对这些选中的记忆路径，进行**一次性加强**：提高路径上每个节点的核心度，以及每条边的正反向强度。

**为什么要结合身份职责？**
- 避免强化那些虽然存在但与该AI无关的记忆（浪费资源，甚至可能干扰）。
- 让AI更专注于它应该擅长的领域，形成**职业化的长期记忆**。
- 模拟人类“干一行，精一行”的特点——一个厨师不会天天去强化数学公式。

**重放时机的建议**：
- 每次成功完成一个与核心职责相关的任务后，立即对该任务的思维链进行重放强化。
- 或者在每日睡眠整理时，按职责相关性排序，优先重放得分高的记忆路径。



### 七、强化：每一次思考都在雕刻记忆

当你成功完成一次推理或任务时，你走过的思维路径——也就是经过的节点和边——都会得到强化：

- 路径上的每条边（正向和反向）强度增加。
- 路径上的每个节点核心度增加。
- 路径开始的那个节点，强化幅度最大（因为它是思考的起点）；越往后，强化幅度递减。

这符合“熟能生巧”的直觉：你越是从A想到B，这条联想就越快、越牢。

---

### 八、情绪标记：成功与失败的额外影响

如果某个思考或行动得到了“正向反馈”（例如解决了问题、用户满意），那么整条路径上的记忆会获得一次额外的增强，幅度大于普通使用。  

如果存在新的方案，就会在原本的流程旁边添加新的知识点；如果旧的方案某一个步骤需要修改，就在旁边添加新方案，弱化错误方案连结（可以作为备选），强化正确的连结。最近正确的连结，必须比最近错误的连结强度高。

反之，如果得到了负向反馈（例如出错、用户不满），相关记忆会受到轻微的削弱。

这模拟了人类“记住成功经验、淡化失败教训”的倾向。

---

### 九、记忆的整理与清理

长期积累，记忆中难免会有冗余或不再有用的内容。因此需要定期进行“整理”：

1. **合并相似节点**：如果两个节点的意思高度重复，就把它们合并成一个，保留更强的核心度和联想边。
2. **删除无用节点**：如果一个节点所有关联边的强度都极低，且自身核心度也很低，很久没有被访问过，就可以彻底删除——腾出空间，让重要的记忆更清晰。

整理过程类似于人的睡眠：白天经历过的事情，晚上大脑会回放、巩固、丢弃不重要的细节。

---

### 十、外部文档：记忆的延伸

不是所有东西都要放进联想网。对于那些内容庞大（如一本手册）、不常使用、或者只需要留作记录的信息，可以存为外部文档。联想网中的节点只需保留一个指向外部文档的指针（比如文件名或链接）。

当需要时，再单独读取文档内容。这样，记忆网络始终保持轻量和高效。

---

### 十、设计哲学总结

- **方向不对称**：联想天然有强弱方向，但方向可被强化。
- **核心度**：节点本身的重要性独立于联想路径。
- **用进废退**：使用强化，不用衰减。
- **情绪加速**：成功/失败对记忆有额外影响。
- **定期整理**：合并相似，删除无用。
- **外部延伸**：大块内容不入图，只存指针。

这套方案的目标是让AI的记忆更像一个**活生生的人**：会忘、会巩固、会找到最相关的线索、会沿着熟悉的路径联想，也会偶尔尝试逆向探索。它不是静态的数据库，而是一个不断生长的联想网络。

## 记忆系统纠错机制 —— 技术要点总结

### 1. 错误信号来源
- **用户明确纠正**：直接指出“错了”或给出正确答案。
- **任务执行结果**：代码运行报错、工具调用失败、输出不符合预期格式。
- **逻辑一致性检测**：规则引擎或辅助模型检测推理中的矛盾（如与强事实冲突）。
- **时间/资源超限**：思维链过长或反复重试失败，视为低质量路径信号。

### 2. 错误定位方法（沿思维链回溯）
- **记录完整路径**：每次推理保存节点访问序列及经过的边（带时间戳）。
- **嫌疑度规则**：
  - 路径末端节点嫌疑最高（直接导致错误结果）。
  - 与已知强事实冲突的节点优先怀疑。
  - 核心度低或边强度低的节点更可疑（本身不牢靠）。
- **用户指定位置**：若用户指出具体步骤，直接锁定对应节点或边。

### 3. 修改决策分级
| 级别 | 决策者 | 触发条件 |
|------|--------|----------|
| 自动执行 | 系统内置规则 | 错误明确且轻微（如用户指出具体边） |
| 辅助确认 | AI提议 + 用户确认 | 错误位置不明确，或需中等幅度改动 |
| 人工介入 | 用户直接操作 | 重大知识错误、安全相关、结构性问题 |

### 4. 修改方式

#### 4.1 降低强度
- 对出错边（正向/反向）减少当前强度值。
- 单次降幅上限：不超过原强度的 20%（避免矫枉过正）。
- 可同时降低节点核心度。

#### 4.2 添加纠错边
- 创建从错误节点指向正确信息的边。
- 初始强度设为较高值（如 0.7），确保下次触发时能引出纠正内容。
- 原错误边保留但降低强度，长期不被使用则自然衰减。

#### 4.3 标记待验证
- 给节点或边打上 `dubious` 标签。
- 后续检索时降低其优先级（如乘以 0.5 系数）。
- 经过多次验证（成功/失败）后，再决定清除标签或执行其他修改。

#### 4.4 删除节点
- 仅当节点内容完全错误且无修正价值（如纯幻觉事实）时执行。
- 删除前确认所有关联边的强度已低于阈值且长期未使用。

#### 4.5 负面连结（抑制边）
- 引入独立边类型，强度为负值（如 -0.9）或单独 `inhibition` 字段。
- 作用：阻断从 A 到 B 的联想，强制禁止路径。
- 建立条件：用户明确禁止、安全违规、逻辑矛盾。
- 衰减规则：抑制强度也可随时间衰减，但速率远低于正向边（如半衰期更长）。

### 5. 验证闭环（修改效果评估）
- **短期验证**：下次遇到类似问题时，观察修改后的路径是否导向正确结果。
- **长期统计**：定期计算节点/边的“成功率”（成功调用次数 / 总调用次数）。
- **自适应调整**：
  - 若成功率上升 → 固定修改，甚至可进一步增强。
  - 若成功率下降或不变 → 撤销修改或尝试其他修正方案。

### 6. 独立监控器
- 一个轻量、只读的辅助模块（可复用规则引擎或小模型）。
- 职责：
  - 监听思维链输入、输出及中间步骤。
  - 接收外部反馈（用户、执行结果）。
  - 调用一致性检测。
  - 生成修改建议（不直接执行，交给记忆系统主模块）。
- 目的：避免主模型“自己审判自己”，保证纠错客观性。

### 7. 与其他模块的交互
- **遗忘曲线**：错误导致的强度降低与自然衰减叠加；多次出错可加速衰减系数。
- **情绪标记**：负反馈视为负向奖励，触发边/节点强度下调。
- **睡眠整理**：在合并相似节点时，优先处理带 `dubious` 标记的节点；对抑制边进行定期审查（用户确认是否仍需保留）。

### 8. 安全与可追溯
- **操作日志**：记录每次修改的时间、原因、修改前后的值，支持回滚。
- **用户白名单**：某些关键知识（如安全规则）禁止自动修改，必须人工介入。
- **冲突解决**：当纠错边与抑制边同时存在时，抑制边优先级更高（安全优先）。

---

以上要点覆盖了从错误检测、定位、决策、执行到验证的完整技术闭环，可集成到任何图结构记忆系统的实现中。

# AI记忆系统技术设计文档

> 版本：v1.0  
> 基于《AI记忆系统方案构想 (v2.1)》及后续维护讨论  
> 本文档描述系统的技术架构、数据模型、算法、维护策略及接口设计。

---

## 1. 概述

### 1.1 目标
构建一个支持有向联想、动态遗忘、强化学习、纠错及定期维护的图结构记忆系统，为AI Agent提供长期、结构化、可进化的记忆能力。

### 1.2 核心特性
- 有向加权边，正向/反向强度独立
- 基于使用频率和成功率的动态半衰期
- 节点核心度（Coreness）独立于边
- 多级纠错（自动/辅助/人工）
- 支持抑制边（负面连结）
- 后台维护：合并、剪枝、重放、验证
- 并发读写锁
- 操作日志与增量备份
- 可视化用户界面

---

## 2. 数据模型

### 2.1 节点 (Node)
```sql
CREATE TABLE nodes (
    id TEXT PRIMARY KEY,
    content TEXT NOT NULL,
    coreness REAL DEFAULT 0.5,          -- [0,1] 核心度
    tags TEXT,                          -- JSON数组
    doc_ref TEXT,                       -- 外部文档指针
    embedding BLOB,                     -- 向量，可选
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    last_accessed TIMESTAMP,
    access_count INTEGER DEFAULT 0,
    is_deleted BOOLEAN DEFAULT 0        -- 软删除，用于冷存储
);
```

### 2.2 边 (Edge)
```sql
CREATE TABLE edges (
    source_id TEXT,
    target_id TEXT,
    forward_strength REAL DEFAULT 0.5,   -- 正向强度 [0,1]
    backward_strength REAL DEFAULT 0.5,  -- 反向强度 [0,1]
    base_stability REAL DEFAULT 0.5,     -- 基础稳定性（影响衰减）
    last_review TIMESTAMP,
    review_count INTEGER DEFAULT 0,
    relation_type TEXT,                  -- 因果/相似/包含/顺序等
    inhibition REAL DEFAULT 0.0,         -- 负向抑制强度 [0,1] 0表示无抑制
    is_dubious BOOLEAN DEFAULT 0,        -- 待验证标记
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (source_id, target_id),
    FOREIGN KEY (source_id) REFERENCES nodes(id),
    FOREIGN KEY (target_id) REFERENCES nodes(id)
);
```

> 注：`inhibition` 字段表示从 source 到 target 的抑制强度，值越高越应避免该联想。检索时若抑制强度 > 阈值（如0.7），则直接阻断。

### 2.3 操作日志 (OperationLog)
```sql
CREATE TABLE operation_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    operator TEXT,                       -- 'system' / 'user' / 'monitor'
    operation TEXT,                      -- 'create_node', 'modify_edge', 'merge', 'prune' ...
    target_id TEXT,                      -- 节点或边的标识
    old_value TEXT,                      -- JSON
    new_value TEXT,
    reason TEXT                          -- 错误修正原因等
);
```

### 2.4 思维链日志 (ChainLog)
```sql
CREATE TABLE chain_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    chain_json TEXT NOT NULL,            -- 节点ID序列 + 边信息
    success BOOLEAN,
    error_location TEXT                  -- 用户或系统标记的错误节点/边
);
-- 每半年清理一次（由后台任务执行）
```

## 3. 核心算法

### 3.1 当前强度计算
```python
def current_strength(edge, now):
    dt = (now - edge.last_review).hours
    half_life = INIT_HALF_LIFE * (1 + ALPHA * edge.review_count)
    return edge.base_stability * math.exp(-dt / half_life)
```
正向联想强度 = `current_strength(forward)`  
反向联想强度 = `current_strength(backward)`

### 3.2 注意力加载分数
```python
load_score = (BETA * node.coreness 
              + GAMMA * task_relevance 
              + DELTA * avg_outgoing_strength)
```
其中 `task_relevance` 通过节点embedding与用户输入的余弦相似度计算。

### 3.3 强化（成功调用）
```python
def reinforce_path(path_nodes, path_edges, success=True):
    factor = 1.0
    for node in path_nodes:   # 从起点到终点
        node.coreness = min(1.0, node.coreness + 0.01 * factor)
        node.last_accessed = now
        node.access_count += 1
        factor *= 0.8   # 越往后强化越小
    for edge in path_edges:
        edge.base_stability = min(0.95, edge.base_stability + 0.02)
        edge.last_review = now
        edge.review_count += 1
        # 如果边是逆向使用的，也应强化相应的方向
```

### 3.4 衰减（遗忘）
- 每次查询时**实时计算**强度，不修改存储值。
- 强度低于阈值（如0.1）的边在检索时忽略。

### 3.5 纠错与抑制
- **降低强度**：直接修改 `base_stability`，单次降幅 ≤20%。
- **添加纠错边**：创建 `错误节点 → 纠正节点` 边，初始强度0.7，`relation_type='correction'`。
- **添加抑制边**：设置 `inhibition` 值（如0.8），可设置 `last_review` 以支持衰减。
- **标记待验证**：设置 `is_dubious = 1`，检索时强度乘以0.5。
- **删除节点**：软删除（`is_deleted=1`），同时将所有关联边也软删除。

## 4. 维护机制

### 4.1 定期任务（后台调度，空闲时触发）
| 任务 | 触发条件 | 操作 |
|------|----------|------|
| **合并相似节点** | 每天一次 | 计算所有节点embedding相似度 >0.9的节点对合并 |
| **剪枝** | 每天一次 | 删除满足条件的软删除节点和强度极低的孤立节点 |
| **重放** | 每周一次 | 结合Agent身份职责，挑选核心路径增强 |
| **验证** | 每次修改后 | 记录修改，下次使用时统计成功率 |
| **清理日志** | 每半年 | 删除超过半年的 `chain_log` 记录 |

### 4.2 合并相似节点算法
1. 选择候选对（相似度>0.9）。
2. 创建新节点，内容由LLM合并，`coreness = max(c1, c2)`。
3. 将所有指向旧节点的边重定向到新节点，边强度取加权平均（按原边强度加权）。
4. 软删除旧节点。

### 4.3 剪枝条件
```sql
-- 硬删除（彻底删除）
DELETE FROM nodes WHERE is_deleted=1 
  AND (SELECT COUNT(*) FROM edges WHERE source_id=id OR target_id=id) = 0
  AND last_accessed < DATE('now', '-90 days')
  AND coreness < 0.2;

-- 强度极低的孤立节点（软删除后等待下次彻底删除）
UPDATE nodes SET is_deleted=1 
WHERE coreness < 0.1 
  AND NOT EXISTS (SELECT 1 FROM edges WHERE source_id=id OR target_id=id);
```

### 4.4 重放策略
- 读取Agent身份配置文件（如 `agent_profile.yaml`），包含 `roles` 和 `core_tasks`。
- 检索与该身份高度相关的节点（通过 `tags` 匹配），选择核心度 >0.6 的节点路径。
- 模拟遍历这些路径，但不实际执行，只进行强度强化：`base_stability += 0.01`，`coreness += 0.005`。

### 4.5 并发锁机制
- 使用SQLite的 `BEGIN IMMEDIATE` 事务。
- 关键操作（写节点/边）前获取锁，完成后 `COMMIT`。
- 读操作使用 `BEGIN DEFERRED` 或快照隔离。
- 若死锁发生，重试最多3次后报错。

## 7. 备份与恢复

- **全量备份**：每周日凌晨自动执行 `sqlite3 memory.db .dump > backup_$(date).sql`
- **增量备份**：每日备份 `memory.db` 文件到 `backups/incremental/`，保留30天。
- **恢复**：通过可视化界面选择备份文件恢复，或手动执行 `.restore` 命令。

## 8. 错误修正闭环（监控器）

### 8.1 独立监控器组件
- 监听思维链日志，检测失败（基于任务结果或用户反馈）。
- 调用一致性检测模块（规则引擎或小模型）。
- 生成修改建议（如 `{"type": "reduce", "target": "edge:uuid", "value": 0.2}`）。
- 将建议发送给主系统，由主系统根据分级策略执行。

### 8.2 分级策略
| 级别 | 条件 | 执行方式 |
|------|------|----------|
| 自动 | 用户明确指出具体边/节点 | 直接执行 |
| 辅助 | 系统建议，但影响范围中等 | 生成预览，用户确认 |
| 人工 | 影响核心安全或大规模修改 | 仅生成报告，需用户手动介入 |

## 9. 外部文档同步

- 当 `doc_ref` 指向的文件被修改或删除时，系统在下次访问该节点时**记录警告**到操作日志。
- 用户可通过可视化界面更新 `doc_ref` 或重新关联。
- 不自动删除节点，只发出通知。

## 10. 性能指标监控

系统记录以下指标到内部表 `stats`：
- 节点数、边数
- 平均查询延迟（最近100次）
- 维护任务耗时（合并/剪枝/重放）
- 操作日志中的错误频率
- 若单次查询返回节点数 > 50，记录警告。
